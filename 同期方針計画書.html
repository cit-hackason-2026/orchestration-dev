§ 02

システム構成

Physical layout · Role assignment

2.1 物理構成図

I²Cバスを用いて指揮者 Arduino（マスター）から各楽器スレーブへ小節マーカーを配信する。各楽器は個別のProcessingインスタンスに接続され、それぞれ自分の倍音合成とスピーカー出力を持つ。

Note. I²Cバスは指揮者→全楽器への1方向ブロードキャスト。Serial/USBは各楽器→自分専用のProcessingインスタンスへの1対1接続。各Processingは担当楽器の倍音構造に特化した音色合成を行う。

2.2 役割分担

装置役割出力
指揮者 Arduino MASTER全体のテンポ管理、センサー入力受付、SYNCマーカー配信I²C信号
楽器 Arduino A/B/C SLAVE主旋律パート(輪唱の各パート担当)、音符イベント生成Serial (音符データ)
リズム Arduino SLAVE拍を刻むリズム楽器、輪唱には参加しないSerial (音符データ)
Processing HOST音符イベントを受け取り、楽器の倍音構造に基づく音声合成音声出力
§ 03

輪唱構造とは

Round · Canon · Time-shifted harmony

3.1 輪唱の定義

輪唱とは、同じ旋律を複数のパートが時間差で歌い重ねる形式。英語では "round" または "canon"。

ポイント — 各楽器は同じ旋律を演奏している。スタート時刻だけが異なる。時間的なズレによって生まれるハーモニーが輪唱の本質である。

3.4 輪唱であるための条件

楽器間の"小節単位のずれ"が一定であること。

// 常に成立すべき条件
ズレ(小節数) = (楽器C位置) − (楽器A位置)
            = 常に一定  // 例: 4小節

このズレが変動すると、輪唱構造は破壊され、最終的には全員がユニゾンになってしまう。
→ 全楽器が常に同一BPMで進む必要がある。

3.2 時間軸で見る輪唱

きらきら星を3パート輪唱、2小節ずつずらした場合のタイムライン。再生ヘッドを動かすと、同じ時刻にそれぞれのパートが別のフレーズを歌っていることが視覚的に分かる。

§ 04

同期の基本アーキテクチャ
ハイブリッド方式

SYNC markers · Phase-Locked Loop · Self-running timer

指揮者が小節の頭ごとにSYNCマーカーを全楽器に配信する。楽器は小節の間を自前のタイマーで自走し、マーカー受信時に位相をじわっと補正する(PLL的動作)。

4.2 時間軸で見る動作

4.3 PLL的補正の動作

楽器側は「次のマーカーはこの時刻に来るはず」という予測時刻を持っている。実際の受信時刻と予測時刻の差(誤差)を計算し、その誤差の一部を次の小節長に反映させる。

補正係数の効果

係数追従速度音楽的影響評価
0.7速いカクつく可能性過補正
0.3適度滑らか且つ追従良推奨
0.2遅い滑らかだが追従遅保守的

4.4 なぜこの方式なのか

課題ハイブリッド方式での解決策
通信量を抑えたい小節ごとに1回だけ配信(1秒に0.5回程度)
クリスタルドリフトでズレる毎小節の補正で吸収(最大でも1小節分以内に収まる)
Arduinoの処理負荷を抑えたい受信ISRは軽量、メインループは音符スケジューリングに集中
急な同期ズレにも対応したい誤差50ms以上は強制スナップ

メッセージプロトコル

I²C commands · Byte-level specification · Boot sequence

5.1 I²Cメッセージ一覧

Wireライブラリのバッファ(32バイト)に収まる形式で設計。

CMD名称用途ペイロード
0x01 SYNC MAIN 小節頭マーカー global_bar (2B) + BPM×10 (2B)
0x10 START 演奏開始 なし
0x11 STOP 演奏停止 なし
0x12 CONFIG パート設定(起動時1回) entry_offset (1B) + loop_length (1B) + part_id (1B)

§ 06

可変テンポの仕組み

Sensor-driven BPM · Bar-aligned application · PLL follow

6.1 基本方針

全楽器が常に同一BPMで進む（モデル1 : 物理時刻ベース）。
パート別に異なるBPMで進むモデル2は輪唱構造を破壊するため不採用。

6.2 センサー入力 → テンポ反映フロー

STEP 01
センサー読取 移動平均 / チャタリング対策
STEP 02
目標BPM更新 デッドバンドで雑音除去 / target_bpm_x10
STEP 03
滑らかな遷移 100ms毎に current_bpm_x10 を target へ寄せる
STEP 04
小節頭で適用 境界まで待機 / applied_bpm_x10 に確定
STEP 05
SYNC配信 全楽器へ同時に新BPMを送信
STEP 06
楽器が追従 PLL補正で位相を合わせる

6.3 なぜ「小節頭でのみ適用」するのか

小節の途中でBPMを変えると、楽器側の「次のマーカー予測時刻」が大きく狂い、拍がガクッとズレる。

✗ 悪い例 — 小節途中で変更

大きくズレて拍が崩れる。

✓ 良い例 — 小節頭で適用

小節境界で切り替わるため拍は崩れない。

反映遅延: 最大で現在の小節1つ分(120BPMなら最大2秒)。音楽的には十分許容範囲。

6.4 BPM変化幅の制限

一度にBPMが大きく変わると、楽器側のPLL補正が追いつかない。また、音楽的にも不自然。

  • 1小節あたりの変化幅は最大10 BPM程度に抑える
  • 急激な変化が指示された場合は、指揮者側で複数小節に分けて段階的に反映
// 疑似コード — 小節境界でのBPM適用
void applyBPMAtBarBoundary() {
  int16_t diff     = target_bpm_x10 − applied_bpm_x10;
  int16_t max_step = 100; // 1小節 = 10.0 BPM

  if (abs(diff) > max_step) {
    applied_bpm_x10 += (diff > 0) ? max_step : −max_step;
  } else {
    applied_bpm_x10 = target_bpm_x10;
  }
}

6.5 輪唱として破綻しない保証

#条件理由
1BPM変更は必ず小節頭で適用小節の途中で変えると拍が乱れる
2変化幅は1小節あたり10 BPM以下楽器側のPLL補正が追従可能な範囲
3全楽器がSYNCを受信できる通信信頼性1楽器でも受信漏れすると輪唱崩壊

これらを守れば、センサー入力に応じたテンポ可変は技術的にも音楽的にも破綻しない。

§ 07

音声合成の役割分担

Arduino → Event / Processing → Synthesis

7.1 tone()関数を使わない理由

Arduino標準のtone()関数は矩形波しか出力できず、楽器らしい音色が再現できない。本システムでは、実際の楽器の倍音構造を調査し、それに基づいた音色合成を行う。

7.2 役割分担

▶ Arduino 側

音符イベント生成

  • 担当パートの楽譜データ
  • SYNCに従った発音タイミング計算
  • 音符(pitch)・強さ(velocity)・長さ(duration)の生成

▶ Processing 側

音声合成

  • 倍音構造による音色再現(クラリネット / フルート / オルガン等)
  • ADSRエンベロープ制御
  • エフェクト付与
  • 最終出力

7.3 Arduino → Processing 音符イベント形式(案)

NOTE_ON  [pitch_midi] [velocity] [duration_ms_H] [duration_ms_L]
NOTE_OFF [pitch_midi]

詳細フォーマットは後続で設計。

7.4 倍音構造調査の方針(今後の作業)

  • 対象楽器:クラリネット、フルート、オルガン等(暫定)
  • 実際の楽器音をスペクトル解析し、基音に対する倍音の強度比を取得
  • Processing側でこの倍音比に基づいた加算合成を実装
§ 08

Arduino性能上の注意点

I²C · Memory · Timing · Serial
8.1
I²C
I²C通信

Wireバッファは 32バイト。スレーブの onReceive ISR内では Serial.print 等の重処理禁止。フラグ立ててメインループで処理する。

8.2
MEM
メモリ管理

Arduino Uno SRAMは 2KB のみ。楽譜データは PROGMEM に配置し、pgm_read_byte() で読み出す。

8.3
⏱
時刻管理

millis() は約49日、micros() は約70分でオーバーフロー。差分計算は必ず 符号付き整数 で。

8.4
SER
シリアル通信

ボーレート 115200bps以上 を推奨。バイナリ形式で音符イベントを送出。Processing側のバッファ溢れに注意。

オーバーフロー安全な時刻比較

// NG  ──  unsigned 比較はオーバーフロー時に誤動作
if (now >= next_time) { ... }

// OK  ──  符号付きで引き算してから比較
if ((long)(now − next_time) >= 0) { ... }
